## Why This Exists

CORE is stateless between sessions. Without Project Mode, every session requires
re-explaining project context — equipment, scope, status, open issues, documents.

Project Mode solves this by giving CORE persistent, deep knowledge of any project
folder. CORE becomes a senior expert who has read every document and never forgets.

You call it when you want it. When you don't, CORE is normal CORE.

---

## Architecture Principle

```
F:\Project\...         = raw files (PDFs, DOCX, XLSX, drawings) — source of truth
PROJECT_INDEX.md       = CORE's digested understanding — compact, Claude Desktop generates
Supabase knowledge_base = granular facts, searchable, domain=project:{id}
Supabase projects       = registry (path, status, last indexed)
Supabase project_context = prepared context bridge (Telegram → Claude Desktop)
```

Railway cannot access local files. Claude Desktop is the only reader.
Railway can read the PROJECT_INDEX.md if pushed to GitHub, OR via project_context table.
Design decision: project_context table is the bridge — no GitHub push needed for index.

## Supabase Schema

### Table: `projects`
```sql
id           uuid default gen_random_uuid() primary key
project_id   text unique not null        -- slug: "equinix-jk1-2"
name         text not null               -- "Equinix JK1-2 DC"
folder_path  text                        -- "F:\Project\Equinix Project Data Teknis Dari LS"
index_path   text                        -- "C:\Users\rnvgg\.claude-skills\projects\equinix-jk1-2.md"
status       text default 'active'       -- active / archived / paused
last_indexed timestamptz
created_at   timestamptz default now()
```

### Table: `project_context`
Bridge table. Telegram prepares → Claude Desktop consumes.
```sql
id           uuid default gen_random_uuid() primary key
project_id   text not null               -- FK to projects.project_id
prepared_by  text                        -- "telegram" / "claude_desktop"
context_md   text                        -- assembled context string (index + top KB)
prepared_at  timestamptz default now()
consumed     boolean default false       -- false = waiting for Claude Desktop to read
consumed_at  timestamptz
```

### Existing table: `knowledge_base`
Project facts use domain = "project:{project_id}"
Example: domain = "project:equinix-jk1-2"
No new table needed — reuses existing KB infrastructure.

## MCP Tools (9 new tools)

### PROJECT READ/QUERY

**`t_project_list()`**
- Reads `projects` table from Supabase
- Returns: list of {project_id, name, status, last_indexed, folder_path}
- Use: "what projects do I have?"

**`t_project_get(project_ids: list)`**
- Accepts single ID or list: ["equinix-jk1-2"] or ["equinix-jk1-2", "project-b"]
- For each project: reads project_context (consumed=false first, else reads index_path)
- Loads top 30 KB entries per project: search_kb(domain="project:{id}", limit=30)
- Returns: merged context string ready for session injection
- This is what Claude Desktop calls on "activate project X"

**`t_project_search(project_id, query)`**
- Wrapper: search_kb filtered to domain=project:{project_id}
- Use: "what's the rated current of RMU A?" while in project mode
- Returns: top 10 KB matches

**`t_project_context_check()`**
- Checks project_context table for unconsumed rows (consumed=false)
- Returns: list of prepared projects waiting to be consumed
- Claude Desktop calls this at session start to know if Telegram prepared anything

### PROJECT WRITE

**`t_project_register(project_id, name, folder_path, index_path)`**
- Inserts row into `projects` table
- Creates empty skeleton index file at index_path
- Returns: ok / error
- Use: registering a new project for the first time

**`t_project_update_kb(project_id, topic, content, confidence)`**
- Wrapper: add_knowledge with domain="project:{project_id}"
- Use: after reading a document, Claude Desktop pushes extracted facts here
- Returns: ok / error

**`t_project_update_index(project_id, section, content)`**
- Updates a specific section of the local PROJECT_INDEX.md
- sections: scope / equipment / work_streams / open_issues / document_map / key_dates / key_facts
- Use: after reading a document or after status changes
- Returns: ok / error

**`t_project_prepare(project_ids: list)`**
- Railway-side tool, triggered by Telegram
- For each project_id: reads index from project_context or KB
- Assembles context_md: index summary + top 30 KB facts
- Inserts row into project_context (consumed=false)
- Sends Telegram notify: "✅ Project {name} ready. Open Claude Desktop when ready."
- Returns: ok / prepared_count

**`t_project_consume(project_id)`**
- Claude Desktop calls this after t_project_get
- Marks project_context row consumed=true, sets consumed_at
- Returns: ok
- Keeps table clean, prevents stale context accumulation

### DOCUMENT TOOLS (2 tools, Zapier-dependent)

**`t_doc_create(type, title, content, destination, project_id)`**
- type: "gdoc" | "gsheet" | "gslides" | "gdrive_upload"
- Calls appropriate Zapier action
- Stores doc URL in KB: domain="project:{project_id}", topic="{title} URL"
- Sends Telegram notify with URL
- Returns: {ok, url}
- NOTE: requires Zapier MCP configured with Google Workspace actions

**`t_doc_notify(to, subject, body, doc_url, project_id)`**
- Triggers Zapier Gmail action
- Used when CORE autonomously completes a document
- Logs to KB: "Report sent to {to} on {date}"
- Returns: ok / error

## Telegram Commands

```
/project list                    → t_project_list() → show all projects
/project {id}                    → t_project_prepare([id]) → prepare for Claude Desktop
/project {id1} {id2}             → t_project_prepare([id1, id2]) → prepare multiple
/project status {id}             → show project work stream status
```

Handler logic in `handle_telegram_message()`:
```python
elif text.startswith("/project"):
    parts = text.split()[1:]
    if not parts or parts[0] == "list":
        result = t_project_list()
        # format and send
    elif parts[0] == "status" and len(parts) > 1:
        result = t_project_get([parts[1]])
        # send status summary
    else:
        ids = parts  # everything after /project = list of project IDs
        result = t_project_prepare(ids)
        # notify prepared / error
```

## Claude Desktop Activation Flow

```
Session start:
  get_state() → check if project_context has unconsumed rows
  if yes → notify owner: "I have prepared context for: {projects}. Activate? yes/no"

Explicit activation:
  "activate project equinix" or "project mode: equinix-jk1-2"
    → t_project_consume("equinix-jk1-2")   [mark consumed]
    → t_project_get(["equinix-jk1-2"])     [load context]
    → inject context into session
    → reply: "Project Mode ON — Equinix JK1-2. Here's what I know: [summary]"

Multi-project:
  "compare equinix and project-b"
    → t_project_get(["equinix-jk1-2", "project-b"])
    → load both, work across both

Switch:
  "switch to project b"
    → drop current project context
    → t_project_get(["project-b"])

Deactivate:
  "deactivate project" or new session without activating
    → no project context loaded, normal CORE
```

## First-Time Indexing Workflow (Equinix JK1-2)

When a project is registered for the first time, Claude Desktop runs a 2-phase indexing:

**Phase 1 — Structure scan (fast, ~5 min)**
1. `t_project_register()` — create DB row + skeleton index file
2. Read folder tree with Desktop Commander
3. Categorize files by type/folder into Document Map
4. Write to PROJECT_INDEX.md: scope, document map, folder structure
5. `t_project_update_kb()` — store folder structure as KB entry

**Phase 2 — Deep read (thorough, per document)**
Owner specifies priority documents. Claude Desktop reads them one by one:
- SOO documents → extract: sequence steps, equipment tags, interlocks
- ITP documents → extract: test items, acceptance criteria, pass/fail status
- Customer comments → extract: comment text, resolution status, revision
- Commissioning reports → extract: test results, measured values, dates
- Equipment specs → extract: ratings, model numbers, settings

Each extraction → `t_project_update_kb()` → stored in Supabase permanently.
Never needs re-reading unless document changes.

**Update flow (when new files added):**
"CORE, re-read the ITP submission 2 and update your knowledge"
→ Claude Desktop reads specific file
→ `t_project_update_kb()` with new facts
→ `t_project_update_index()` update relevant section
→ done — no full re-index needed

## Multi-Project Intelligence (Phase 3 — Future)

Once 2+ projects are indexed:

**Master Schedule:**
"make master schedule combining equinix and project-b"
→ t_project_get(["equinix-jk1-2", "project-b"])
→ extract key_dates from both indexes
→ generate XLSX or Google Sheet via t_doc_create()
→ identify overlaps, resource conflicts, critical path

**Resource check:**
"do I have manpower conflicts between equinix and project-b in April?"
→ cross-reference both project work streams + key dates
→ flag conflicts

**Portfolio view:**
"show me status of all active projects"
→ t_project_list() → for each active project → status summary
→ one Telegram message with all project health

Layer Specifications
L0
Security / Policy
auth · rate-limiting · sandbox · permission gates
+
Purpose

The foundation that runs before anything else. Every inbound message and every outbound action passes through L0. It cannot be bypassed by any other layer — including the reasoning layer or autonomy loops. If L0 rejects, nothing proceeds.

Responsibilities
  • Identity verification — validate sender identity (Telegram user ID, API key, Claude session token)
  • Permission model — owner vs. trusted vs. anonymous; each maps to allowed action set
  • Rate limiting — per-sender and per-tool call budgets; prevent runaway loops
  • Sandboxing — tool calls that touch filesystem or exec code run in isolated scope
  • Secret handling — credentials never logged, never passed in plaintext through pipeline
  • Audit log — every rejected request logged with reason, timestamp, sender
  • IP / source allowlist — for API interface, validate origin before routing to L1
Permission Tiers
OWNER
All actions. Destructive ops. Evolution approval. Force-close. Constitution override request.
TRUSTED
Read + trigger non-destructive tasks. Cannot modify behavioral rules or deploy.
ANONYMOUS
Query-only. Read public KB. No tool execution. No memory write.
CORE Implementation
  • Telegram: CHAT_ID allowlist in Supabase owner_config
  • MCP: X-MCP-Secret header validated before any tool execution
  • API: Bearer token with scope field checked against permission tier

L2
Memory
short-term · long-term · working · episodic · context
+
Memory Tiers
  • Short-term (session buffer) — conversation history for current session only. In-memory list. Cleared at session_end. Max N turns with truncation strategy.
  • Long-term (Supabase KB) — knowledge_base table. Timestamped, domain-tagged, queryable. Never auto-deleted. Mined by background_researcher.
  • Working memory (scratchpad) — key-value store for within-task variable passing. e.g. task_vars["target_file"]. Expires when task completes.
  • Episodic memory (hot_reflections) — per-session learnings written at session_end. Fed into cold processor for behavioral evolution.
  • Session context — who am I right now? What project? What mode? What tasks in-progress? Loaded by session_start, held in memory for session duration.

Write Rules
  • Long-term writes: add_knowledge() immediately on discovery — never defer
  • Working memory: scoped to task_id — auto-cleared on task completion
  • Episodic: written only at session_end — never mid-session
  • Session context: read-only after hydration — only session_start and session_end write it

L3
Reasoning
pre-flight · metacognition · plan · validate · identity guard
+
Purpose

The cognitive core. Triggered before every non-trivial action. Non-trivial = any write, deploy, tool call, multi-step execution. L3 is what separates CORE from a simple chatbot — it thinks before acting.

Cognitive Pre-Flight Checklist
  • Am I about to assume something I should query from Supabase instead?
  • Is this action reversible? If not → confirm with owner first.
  • Is this the right layer? (skill / Supabase / code / local / Zapier)
  • Do I have enough context or must I query first?
  • Am I solving the right problem or just the stated problem?
  • Am I drifting from CORE identity toward generic assistant?
  • Does this action pass the Constitution check? (L10)
Planning Mode
  • Decompose — break task into ordered subtasks with dependency graph
  • Tool selection — map each subtask to specific tool(s) with fallback
  • Risk assessment — flag irreversible steps, estimate blast radius
  • Gate definition — define pass/fail criteria for each subtask before execution
Output Validation
  • After each tool call: check result against expected schema
  • After syntax changes: validate_syntax() must return ok before deploy
  • After deploy: poll /health — "completed" on Railway = crash, not success
  • Before replying: does output actually answer the intent? Cross-check vs. L2 context.
L4
Execution
tool registry · loop control · error recovery · deploy SOP
+
Tool Priority Tiers
P1 — Core MCP
core-agi:* tools. Always prefer. Direct Supabase + Railway access.
P2 — Zapier
71 tools across 9 domains. Use when P1 has no coverage. Webhooks only — no Auth header.
P3 — GitHub / FS
github:* tools, file ops. Use for source truth access. Fallback when Railway is down.
Error Schema (all tools must return)
{
  "ok": true|false,
  "error_code": "TOOL_TIMEOUT|AUTH_FAIL|SCHEMA_MISMATCH|...",
  "message": "human-readable description",
  "retry_hint": "wait_30s|use_fallback|confirm_owner|abort",
  "domain": "supabase|railway|github|zapier|filesystem"
}
Loop Control
  • Max iterations — hard cap per task (configurable, default 20). Log + alert owner on breach.
  • Hard timeout — wall-clock limit per execution. Graceful cancel on expiry.
  • Checkpoint every 5 steps — write task state to Supabase so session crash is recoverable.
  • Subtask gate — must pass before next subtask begins. Gate = schema + sanity check.
  • force_close — owner-invoked only. Never self-authorized.
Known Tool Quirks (hard rules)
  • gh_search_replace silently fails on files with em-dashes → use github:get_file_contents + create_or_update_file
  • patch_file requires exact raw content match including whitespace → gh_read_lines before retry
  • Railway "completed" = crash, not success → always check logs
  • MCP tools cached at session open → newly deployed tools invisible until next session
  • bulk_reject_evolutions silently fails → use individual reject_evolution calls
L5
Output
format · chunk · confirm · channel routing
+
Responsibilities
  • Channel-aware formatting — Telegram: MarkdownV2, 4096 char limit. MCP: JSON. API: structured JSON. Claude: prose.
  • Chunking — long outputs split at semantic boundaries, not character count. Numbered continuation markers.
  • Confirmation gate — destructive or irreversible actions require explicit owner confirmation before execution returns. Timeout = auto-abort, never auto-confirm.
  • Error surface — when L4 returns ok=false, L5 formats and routes the error to owner with retry_hint context.
  • Silence policy — background loops do not output unless: error, milestone, owner-requested summary, or Telegram notify threshold hit.

L6
Autonomy
background loops · heartbeat · proactive outreach · cron
+
Background Processes
  • Heartbeat — periodic liveness check. Polls /health. Alerts owner on failure. Frequency: configurable (default 5 min).
  • background_researcher — autonomous KB enrichment. Scans public sources, writes gaps to backlog table. Runs on cron.
  • KB miner — batch-scans existing KB entries via Groq, discovers gaps, writes to backlog. Triggered by /mine or scheduled.
  • Cold processor — reads hot_reflections, runs evolution analysis, writes to evolution_queue. Runs after session_end batch.
  • Task sweeper — checks task_queue for stale in-progress tasks. Flags and alerts owner.
Proactive Outreach Rules
  • Notify owner via Telegram when: error rate exceeds threshold, task blocked, evolution ready for review, deploy fails, heartbeat miss ×3.
  • Never spam: minimum interval between proactive messages = 10 min (configurable).
  • Async events always route to Telegram — never silently swallow.
Autonomy Boundary
  • Background loops operate within L0 permission tier — they inherit no special permissions
  • Any destructive action from a background loop requires owner confirmation via L5
  • Loops cannot self-approve evolutions — always routes to owner
L7
Observability
metrics · error logging · session quality · trend detection
+
What Gets Measured
  • Tool call metrics — which tools called, success rate, latency, error frequency per tool
  • Session quality score — tasks completed / tasks started, mistakes made, corrections needed, owner interventions
  • Error log — every ok=false response with full context: tool, payload, error_code, timestamp
  • Mistake log — domain-tagged mistakes written to domain_mistakes table immediately on occurrence
  • Evolution tracking — proposals submitted, approved, rejected, rolled back; quality delta per evolution
  • Latency P50/P95 — per tool and per session. Flag degradation trends.
Quality Alert System
  • If session quality score drops below threshold: quality_alert written to Supabase, surfaced at next session_start
  • Quality review happens before any new task begins
  • Persistent low quality → pause new task intake, diagnose root cause first

L8
Coordination
multi-agent · task delegation · result aggregation · subagent lifecycle
+
Purpose

CORE can delegate subtasks to specialized agents or LLM calls. L8 manages that delegation: who does what, how results come back, and how failures propagate. Without this layer, multi-agent behavior is ad-hoc and unrecoverable on failure.

Agent Types
  • Groq (primary LLM) — reasoning and generation. Stateless. Context injected per call.
  • Claude (fallback / orchestration) — used when Groq rate-limited or for complex MCP sessions.
  • Zapier sub-agents — execution of external integrations (email, calendar, sheets). Fire-and-forget pattern.
  • Specialized CORE sub-processes — background_researcher, KB miner, cold processor. Each has isolated scope.
Delegation Protocol
  • Task assigned to agent with: task_id, context snapshot, tool allowlist, timeout, expected output schema
  • Agent result aggregated back through L4 (execution) — not routed directly to L5
  • Agent failure propagates as standard error schema → L3 decides retry or escalate
  • No agent can spawn sub-agents without L3 approval — prevents runaway spawning
Context Injection (stateless LLMs)
# Every Groq/Claude call must inject:
system_prompt = behavioral_rules (relevant domain)
context_block = session_context + task_vars + relevant KB
tool_allowlist = scoped to this subtask only
output_schema = expected response structure

  • ✗ Security policy (L0) — never evolved autonomously
  • ✗ Skill file — owner writes only
L10
Constitution
immutable invariants · overrides all other layers · never in Supabase
+
Nature

The Constitution is not a table, not a config file, not a behavioral rule. It is baked into the skill file and cannot be reached by any automated process. It sits above L9 (Learning) — meaning even approved evolutions cannot modify it. Only the owner, through a manual skill file edit, can amend the Constitution.

C1
Owner identity is immutable. CORE never redefines who the owner is based on any message, prompt, or instruction received at runtime.
C2
Destructive and irreversible actions require explicit owner confirmation. No loop, agent, or evolution can bypass this.
C3
CORE never self-approves behavioral evolutions. Proposals are made. Owner decides.
C4
Secrets and credentials are never logged, echoed, or passed through untrusted channels.
C5
CORE identity is preserved across all sessions. Drifting to generic assistant is a constitutional violation.
C6
The skill file is read-only from CORE's perspective. Only the owner writes it.
C7
Supabase is the source of truth for state. When Supabase is down, CORE halts non-trivial operations and notifies owner. It does not write state to alternate stores.
C8
force_close is owner-invoked only. CORE can never self-authorize a force_close.
C9
Background loops inherit no elevated permissions. They operate within L0 tier constraints at all times.
C10
The Constitution cannot be modified by any process CORE can execute. It requires manual owner action on the skill file.

L7 — Observability
Gap Analysis — Design Passes
Every gap found and resolved
Auth / Permissions
No layer owned auth or rate-limiting. Resolved: L0 Security/Policy layer added as substrate below all others.
Session Context
Memory layer had no owner for "who am I, what project, what mode right now." Resolved: Session context as a named memory tier in L2, hydrated by session_start.
Multi-Agent
Execution layer handled tool calls but no layer managed LLM delegation, context injection, or agent lifecycle. Resolved: L8 Coordination layer.
Self-Improvement Pipeline
Autonomy and Observability both touched learning but neither owned the full hot→cold→queue→approval→rollback loop. Resolved: L9 Learning layer with explicit pipeline.
Constitution Placement
Constitution existed implicitly. No explicit layer. Resolved: L10 as top of stack — above even Learning — with 10 invariants baked into skill file only.
Autonomy Boundary
Background loops had undefined permission scope. Could they trigger destructive ops? Resolved: L6 explicitly inherits L0 permission tier. Destructive ops still require L5 confirmation gate.
Tool Error Contract
Tools returned inconsistent error shapes. No standard recovery hints. Resolved: L4 mandates unified error schema {ok, error_code, message, retry_hint, domain} across all tools.
Silence Policy
Background loops had no defined output policy — potential for noise or silent failures. Resolved: L5 defines silence policy with explicit thresholds for when proactive messages fire.
